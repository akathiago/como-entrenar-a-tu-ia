## 3.4 · Deploy: tu URL real

- **Tiempo:** 15 minutos

> **Lección interactiva:** escribí `/clase-3-4` en Claude Code

## Qué es el deploy

Deploy es llevar tu proyecto de tu computadora a internet. Tu código pasa de tu máquina a servidores que cualquiera puede visitar, con una URL real.

Es EL momento. Todo lo anterior era ensayo: esto es el estreno.

## Qué es el hosting

Servidores que corren tu proyecto las 24 horas y atienden a los visitantes. No los administrás vos: el servicio de hosting se encarga de todo. Vos solo entregás el código.

## Dónde publicar

Hay varios servicios y son parecidos: gratis para proyectos personales, se conectan a GitHub y republican solos cuando subís un cambio.

| Servicio | Por qué |
|----------|---------|
| **[Vercel](https://vercel.com)** | El que usamos por defecto. Hecho por los creadores de Next.js, que es con lo que construimos. |
| **[Netlify](https://netlify.com)** | Funciona igual de bien, el paso a paso es casi idéntico. |
| Otros | Cualquiera que conecte con GitHub sirve. No te cases con ninguno. |

## El paso clave: importás vos el repo

Este paso lo hacés **vos, con el mouse, en el navegador**. No es un capricho: importar el repo es lo que deja tu proyecto **conectado a GitHub**, y esa conexión es la que hace que después se republique solo. Si el deploy se hace por otro lado, esa conexión no existe y el auto-deploy no funciona.

1. Entrá a **vercel.com/new** (o a **app.netlify.com**).
2. **Iniciá sesión con GitHub**, con la misma cuenta donde subiste el proyecto.
3. Dale **permiso para ver tus repositorios**. Tu repo es privado: sin este permiso no lo va a encontrar. Podés darle acceso a todos o solo a este.
4. **Buscá tu repo en la lista e importalo.**
5. No toques la configuración: viene detectada sola. **Deploy.**
6. Esperá el minuto que tarda en construir.

Y sale la URL. Esa es tuya.

## La magia del auto-deploy

Como el repo quedó conectado, de acá en adelante:

1. Cambiás algo en tu proyecto
2. Le pedís a Claude que lo suba a GitHub
3. El servicio detecta el cambio y republica solo
4. Tu sitio se actualiza en un minuto

Sin pasos manuales, sin volver a entrar al dashboard. Cambio → push → está online.

## Problemas típicos

| Problema | Solución |
|----------|----------|
| Mi repo no aparece en la lista | Es el permiso: volvé a autorizar el acceso a los repositorios. Es privado, por eso no se ve solo. |
| Build failed | Copiale a Claude el log de error del dashboard: casi siempre lo arregla solo. |
| El sitio muestra la versión vieja | Está terminando de construir. Esperá un minuto y refrescá. |
| Funciona en localhost pero no online | Pedile a Claude que revise los build logs del dashboard. |
| Subí un cambio y no se republicó | El repo quedó sin conectar. Revisá en el dashboard que el proyecto muestre tu repo de GitHub como origen. |

## Después del estreno

1. **Probalo en el celular**: la misma URL anda en cualquier dispositivo
2. **Compartilo**: mandá el link al grupo del taller, a tu familia, a quien quieras
3. **Actualizá cuando quieras**: cambio → push → se publica solo

## El loop completo, ahora tuyo

| Paso | Qué significa |
|------|---------------|
| **Planificar** | Tener claro qué construís (REQUISITOS.md) |
| **Construir** | Claude lo crea |
| **Iterar** | Afinar con capturas hasta que te guste |
| **Guardar** | Respaldar en GitHub |
| **Publicar** | Importar el repo en el hosting, URL real |

Este circuito sirve para cualquier proyecto de cualquier vertical. Ya no es teoría: lo acabás de hacer.
